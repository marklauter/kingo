# Rewrite interpreters — Contains and Expand

## Clean-room constraint

Design this in a fresh context from this note, the Zanzibar paper (`docs/5068.pdf`), and the current domain code. **Do not read the archive branches' ACL implementation** (`src/Kingo.Acl/` on `main-archive` / `dictionary-encoding`) before or during design — it is incomplete, and its lessons are already encoded below as requirements (Mark, 2026-07-15).

## The work item

Two evaluators over the `SubjectSetRewrite` algebra, pure domain code in `Kingo.Closures` — the one project that references both `Kingo.Graphs` and `Kingo.Schemas`, and the home of the fact-lookup port its adapters implement later:

- **Contains** — the membership predicate behind the Check API: subject ∈ subjectset? A short-circuiting boolean walk; the hot path.
- **Expand** — materialization of one relationship's rewrite tree, single-level wrt indirection (referenced subjectsets stay leaves; see Settled); the audit/debug/UI view.

One algebra, two interpreters. This is the first joint consumer of both aggregates — rules from `Schema`, facts from the graph — so expect it to pressure-test the algebra's semantics the way SDL pressure-tested the names ([[the-first-consumer-forges-the-domain]]).

## Naming

- The domain predicate is **`Contains`** — set-first: `subjectSet.Contains(subject)` matches the ∋ reading ([[domain-language]]) and carries .NET's strongest prior (`set.Contains(member)`). Not `IsMember`/`IsAMemberOf`, which read member-first and re-invert that reading.
- **`Check`** stays the service/API-level name — Zanzibar's reach word ([[four-service-split-by-load-profile]]).
- **`Kookie`** is the Kingo name for the zookie (decided 2026-07-16): an opaque kernel value naming a point in the store's timeline. Request-side it's a *floor* ("no earlier than this snapshot"); Decision-side it's the *pin* ("evaluated at this snapshot"). "Zookie" survives only as the paper-facing alias ([[storage-versioning-design]]).
- **`Kingo.Closures`** is the interpreter project (decided 2026-07-18) — "Acl" names the data structure ReBAC generalized away, and the interpreter answers membership in the derived *closure*: the set closed under the rewrite rules' derivation, graph theory's transitive closure. "Reachability" was rejected: `!` and `&` make the derived set non-monotone, so the verdict is a function over several reachability questions, not a reachability property itself.

## Settled — domain-modeling interviews, 2026-07-16 through 2026-07-18

Kernels pinned with Mark before the clean-room session; these are constraints, not suggestions.

- **`Contains` answers one question: does the graph's derived closure contain this fact.** There is no "deny" concept in the algebra — `!` is set exclusion, just another operator. Membership at a snapshot, nothing else; "authorized" is the Check host's interpretation of the answer, not the interpreter's concern. Both truth values are first-class `Decision`s (clarified 2026-07-17): a completed evaluation issues a Decision whether the verdict is true or false — "not included" is the false verdict, fully auditable and replayable, not an absence. "Denied" is the host's reading of a false verdict (and, under fail-closed policy, of a failure — but that denial has no Decision, only the error event).
- **The question is a putative `Fact`; everything else is context.** `(SubjectSet, Subject)` is exactly a Fact's content, so the signature is `Contains(Fact fact) → Result<Decision>`: does this statement hold at this snapshot — stored directly or derived through the rewrite algebra (the paper's own modality: check takes a "putative" member). `Fact` now carries `(SubjectSet, Subject)` positionally (reshaped 2026-07-16). The evaluation context — the `Schema` value, the snapshot-pinned fact port, the `TimeProvider` — is construction state, not per-call args. The zookie is consumed at the host edge to pin the port; the interpreter never sees a zookie argument.
- **A subjectset-valued member means literal element membership, not subset reasoning.** The `Subject` DU admits `SubjectSet` (a stored `doc:x#viewer@group:eng#member` is legal), so `Contains(doc:x#viewer @ group:eng#member)` asks whether that subjectset appears as an element of the closure — value equality at the leaves, no second semantics under `!`. Matches Zanzibar: Expand's leaves are "user IDs or usersets."
- **`Decision` carries exactly one Fact — the one judged — and none of the facts consulted to judge it.** Decision = the Fact judged, the verdict, the snapshot pin (`Kookie`), the schema version, the wall timestamp. Reproducibility comes from the store's multi-version history, not the payload.
- **The schema version slot is an opaque `SchemaVersion` kernel value** (decided 2026-07-18) — minted as a stub (`src/Kingo/SchemaVersion.cs`), the same move as `Kookie`; its encoding (hash, revision, name plus revision) is [[storage-versioning-design]]'s to fill. It arrives as evaluator constructor context beside the injected `Schema` value.
- **Empty operator nodes are unrepresentable** (decided 2026-07-19): `UnionRewrite`/`IntersectionRewrite` refuse empty operand lists at construction ([[namespace-create-validation]]). The SDL grammar cannot produce the shape; the Write API path could, and the conventional reading of an empty intersection — the universal set, everyone a member — is an authorization hazard, so the shape is refused rather than given semantics.
- **The depth bound is evaluator constructor config** (decided 2026-07-16; Contains only — Expand never recurses and takes none, decided 2026-07-18) — injected context alongside the port and clock, not SDL and not per-call. `DepthExceeded` carries the bound it hit, so audit distinguishes "cycle in data" from "bound too low."
- **Operator error semantics are Kleene: the result is a function of operand values, never of evaluation order** (decided 2026-07-16). Error is the third value; an *absorbing* value dominates it: `false` absorbs in `∩` (false & anything = false), `true` absorbs in `∪` (true | anything = true), and for `a ! b` (= `a ∩ ¬b`) both `a = false` and `b = true` absorb to `false`. Short-circuit is legit exactly when the value in hand is absorbing — the unobserved operand's outcome (true, false, or error) cannot change the answer, so skipping it or cancelling its in-flight parallel evaluation is sound. An error surfaces as `Result.Failure` exactly when it is *decisive*: nothing absorbed it, so the answer genuinely depended on the broken operand. The special case already ruled — `a ! b` with `a` true and `b` errored → `Result.Failure`, no Decision — is this rule. Sequential, reversed, and parallel-with-cancellation evaluation all yield the identical verdict; strict error-poisoning was rejected because it is incoherent with short-circuiting (the verdict would depend on scheduling). The residual risk — a broken graph region hiding behind healthy absorbing branches, surfacing only intermittently — is assigned to the Write path and offline validation (schema cycles at SDL parse, data cycles at write or sweep), not to the hot path.
- **No `Decision.Empty`** (decided 2026-07-17). `Result<Decision>` failure carries no value property at all — the no-answer case is the failure channel itself, so an empty variant has nothing to represent. The general instinct stands elsewhere: where a domain ever needs a no-answer value, prefer a domain singleton (collection-style `Empty`) over null.
- **Tupleset traversal: exactly specified traverses; anything else is a modeled error** (decided 2026-07-16). Evaluating `(parent, viewer)` on `folder:x` reads the facts under `folder:x#parent`; each member is one of three shapes: (1) `folder:y#...` — `SubjectSet` with `RelationshipIdentifier.Nothing`, *exactly specified*: hop 2 substitutes `viewer` for the `...` and recurses at `folder:y#viewer`; (2) `folder:y#viewer`/`#editor` — `SubjectSet` with a real relationship, *over-specified*: modeled error; (3) `folder:y` — bare `DirectSubject`, *under-specified* (no resource+`...` slot to substitute into; `folder:x#parent@user:anne` is the same shape — a legal fact, nonsense for traversal): modeled error, distinct code. Both errors join family 2 as ill-shaped-for-traversal data defects. Silent skipping was rejected (authorization outcomes must not change invisibly with data shape); override semantics for (2) were rejected (turns a data oddity into a permission redirect the paper deliberately avoided). The guard belongs upstream: the Write path refuses wrong-shaped members on relationships any rewrite uses as a tupleset's first element, making the evaluator's error a never-in-practice backstop. That guard is the Write service's work item, not this one; until it exists, conditions 5–6 are the only protection, which is acceptable because they are modeled errors, not crashes.
- **The error taxonomy is settled** (decided 2026-07-16) — three families, eight conditions; see [Error conditions](#error-conditions). Conditions 1–6 are distinct modeled errors owned by `Kingo.Closures`; 7–8 flow through as the port's own error values, propagated untranslated (a wrapper adds a layer without adding a fact). All six are `Result.Failure` per the settled channel. In Contains the Kleene rule decides whether a given evaluation surfaces or absorbs them; in Expand every error is decisive (conditions 3 and 4 are Contains-only — see the Expand rulings).
- **The pinned port exposes its snapshot token, and the token is the `Kookie` type itself** (decided 2026-07-19) — no distinct port token, no mapping layer. An opaque value the interpreter copies into the `Decision` without interpreting, so the Decision is complete at birth — no host-side enrichment. `Kookie` is minted as an opaque kernel stub (`src/Kingo/Kookie.cs`); encoding is [[storage-versioning-design]]'s to fill.
- **The result channel is `Result<Decision>`** (Contains) **and `Result<Expansion>`** (Expand). Undefined namespace/relationship and depth-exceeded are `Result` failures — where no membership answer exists there is no `Decision`, not a Decision wearing an error flag; likewise where no complete tree exists there is no `Expansion`. Fail-closed (error → deny + alarm) is host policy, visible at the edge. Audit covers both channels: success serializes the Decision; failure serializes the error with the request envelope.
- **`Decision` carries two clocks, distinctly:** the snapshot token (logical time — *which* facts) and the wall timestamp from the injected `TimeProvider` (*when* evaluated).
- **`Expand` is single-level with respect to indirection — a graph walk of one step, edge choice left to the caller** (decided 2026-07-17; paper §2.4.5 adopted verbatim). One call materializes one relationship's full rewrite tree — the union/intersection/exclusion structure with direct subjects at the leaves — and referenced subjectsets stay unexpanded leaves; the caller re-expands the leaves it chooses to walk, so transitive closure is the caller's loop, paced edge by edge. Full closure (recurse until only direct subjects remain) was rejected: unbounded fact-driven work per call — the same recursion problem Contains bounds — and resolving `!` leaves drags Expand into re-implementing Contains. A depth-limit parameter was rejected: every consumer must handle partially-expanded trees anyway, so it is the single-level contract with the closure cost profile smuggled in, and it takes whole-level steps where single-level lets the caller decide per edge.
- **Caller identity is envelope, not `Decision`** (decided 2026-07-17). The criterion: a field earns its place on `Decision` iff replaying the decision requires it (Fact, Kookie pin, schema version) or records the judgment's own provenance (verdict, wall timestamp). Caller identity is neither — the same question from a different caller is the same judgment — so it lives in the host's request envelope, and the audit event is envelope + serialized Decision (or envelope + error). The interpreter never sees who asked.
- **`Expand`'s question is a `SubjectSet`** (decided 2026-07-18): `Expand(SubjectSet subjectSet) → Result<Expansion>`. The resource#relationship pair names exactly one relationship's rewrite tree, which is what single-level materializes; no `Subject` in the question — Expand asks "who's in this set," not "is this one in it," so Contains's putative-Fact move doesn't transfer. The result is the tree projection: leaves are subject IDs or subjectsets naming other (resource, relationship) pairs; intermediate nodes are union/intersection/exclusion. The optional kookie is the Read+Expand host's request field (the paper's Expand API), consumed at the edge to pin the port — the interpreter sees only the pinned port, same as Contains. Evaluation context (Schema, pinned port, TimeProvider, applicable bounds) is constructor state.
- **`Expand` never recurses; depth exceeded (condition 3) is Contains-only** (decided 2026-07-18). Paper §2.4.5: Expand "follows indirect references expressed through userset rewrite rules" — the following is contrasted with Read and refers to the schema's rewrite rules, finite and known at parse; fact-side indirection, the only source of unbounded recursion, terminates as leaves ("usersets pointing to other ⟨object#relation⟩ pairs"). Work per call is bounded by schema shape × fact count at one level, never by chain length, so Expand's constructor takes no depth bound.
- **Tupleset conditions 5–6 apply to `Expand` unchanged** (decided 2026-07-18). The tupleset arm cannot stay a pure leaf — without reading the facts under `resource#tupleset-relationship` there is no pair to point a leaf at — so Expand performs the same single fact read Contains does and substitutes to emit leaves; only the recursion into them differs. Same three member shapes, same modeled errors, no silent skip: a tree that silently dropped a parent is the same invisible-outcome hazard the Contains ruling rejected.
- **Condition 4 (undefined relationship mid-walk) is Contains-only** (decided 2026-07-18). It arises exactly when the walk crosses into another relationship's definition; Expand never crosses — both places facts introduce dangling references (subjectset-valued members, tupleset substitution results) terminate as leaves, emitted as data without schema lookup. A dangling leaf surfaces as family 1 on the caller's re-expand, where it *is* the question. Emitting it is the correct audit behavior: the tree shows what the data contains.
- **Every error `Expand` meets is decisive — whole-call failure, no partial trees** (decided 2026-07-18). Kleene absorption exists because a boolean operand can dominate; a tree has no dominating value — every node is part of the answer, so a broken branch always changes it. Partial trees with error-marker nodes were rejected per the settled channel discipline: where no complete answer exists there is no result value wearing an error flag. Expand's error surface is conditions 1, 2, 5, 6, 7, 8; conditions 3 and 4 are Contains-only.
- **`Expansion` is the sibling result type — same five-slot shape as `Decision`** (decided 2026-07-18). The replay-sufficiency criterion transfers verbatim: `Expansion(SubjectSet, Tree, Kookie, SchemaVersion, Timestamp)` — question, answer, pin, schema version, wall timestamp — with `SubjectSet`/tree in the seats `Fact`/verdict occupy on `Decision`. Complete at birth, no host-side enrichment; caller identity stays envelope; two clocks, distinctly. The tree's leaves are subject IDs or subjectsets naming other (resource, relationship) pairs; intermediate nodes are union/intersection/exclusion. The tree DU's shape is design work, not a ruling.
- **`Decision` does not carry the proof path.** The witness doesn't survive `!`: a true verdict under `(this | (parent, viewer)) ! banned` embeds a non-membership claim for `banned`, and non-membership has no path — its proof is the exhausted search (a *false* verdict, conversely, can have a concrete path through `banned`). A path also costs hot-path allocation and leaks other subjects' relationships to any caller allowed to ask yes/no questions. "Why" is recomputed on demand: re-run `Contains` or run `Expand` at the Decision's recorded snapshot — reproducibility is a storage property (multi-version tuples), not a payload property ([[storage-versioning-design]]).

## Error conditions

Three families, eight conditions. Every one is a `Result.Failure` — where no answer exists there is no `Decision` (or `Expansion`). In Contains the Kleene rule decides whether a given evaluation surfaces or absorbs it; in Expand every error is decisive. Expand meets conditions 1, 2, 5, 6, 7, 8; conditions 3 and 4 are Contains-only.

### Family 1 — the question is ill-formed against the schema

Caller defects, detectable before any I/O: the `Schema` is an in-memory value, so both checks are pure lookups that fail before the port is ever touched.

1. **Undefined namespace.** The question's resource names a namespace the schema doesn't define — `Contains(document:readme#viewer@user:anne)` against a schema that defines `doc`, not `document`. There is no rewrite rule to interpret, so the question isn't answerable *false* — it isn't answerable at all: the set the question names doesn't exist in this schema's universe. A denial asserts "evaluated, and not a member" and is reproducible at the snapshot; this asserts "no such set." Mapping it to `false` would convert every caller typo and every caller/schema version skew into silent denial — an outage that looks like policy.
2. **Undefined relationship on the question.** The namespace exists but the relationship isn't defined on it — `Contains(folder:x#editor@user:anne)` when `folder` defines `owner`/`viewer`/`banned`/`parent` and no `editor`. Same character as (1), kept distinct because the repair is different (wrong namespace vs wrong relationship) and the drift source is different: relationship vocabularies evolve much faster than namespace sets, so the typical cause is a caller compiled against a newer schema than the evaluator holds. This is requirement 4's boundary: fail-closed applies to *facts*, never to schema drift or caller defects.

### Family 2 — the evaluation couldn't complete

Defects in the graph, or its interaction with the depth bound. These surface mid-walk, after I/O has begun.

3. **Depth exceeded** (Contains only). The walk hit the constructor-configured bound. The unit is the fact-driven re-entry (ruled 2026-07-18): each tupleset hop and each subjectset-valued member expansion consumes one unit; `ComputedSubjectSetRewrite` crossings are free — acyclic by construction (see Deliberately absent), they cannot recurse without consuming a fact. The bound cannot be computed at design time because the recursion is not bound by the schema — the schema's rewrite tree is finite and known at parse — it is bound by the *facts*: a tupleset like `(parent, viewer)` re-enters the walk once per stored parent fact, and subjectset-valued members re-enter it once per nesting level. Chain length is whatever users have written — `folder:a#parent@folder:b`, `folder:b#parent@folder:c`, … arbitrarily long, or `folder:a#parent@folder:b` plus `folder:b#parent@folder:a`, a genuine cycle. The bound is an operational guard against user-writable data; the error carries the bound it hit so audit can distinguish "cycle in data" from "bound set too low."
4. **Undefined relationship encountered mid-walk** (Contains only — Expand emits the dangling reference as a leaf). Same surface as (2), a different defect: schema/data drift. Facts and schema are separately versioned artifacts that drift independently. Under schema v1, `group:eng#member` is valid and `doc:x#viewer@group:eng#member` is written — validated, stored. Schema v2 removes (or renames) `member` on `group`; the fact is still in the store. An evaluation under v2 reaches that member, needs to expand `group:eng#member`, and finds no such relationship: the fact was valid when written, the schema moved. Write-time validation cannot prevent this — the mutation that broke the fact was a *schema* change, and guarding that side means proving no fact anywhere references the removed relationship, a full-graph scan as a precondition of schema DDL. Two honest layers: a *guard* (schema-change validation, or a deprecate-don't-remove policy) in the Write/schema service design, and this *modeled error* in the evaluator — which cannot assume the guard held for all history, especially when evaluating an old snapshot pin against a schema version from a different moment. The seductive alternative — treat the dangling reference as an empty set — was explicitly rejected: it silently changes authorization outcomes under `!`, where an undefined `banned` would mean *nobody is banned*.

5. **Tupleset member over-specified.** Traversal met a `SubjectSet` member carrying a real relationship — `folder:x#parent@folder:y#viewer` — where the exactly-specified shape is `folder:y#...` (`RelationshipIdentifier.Nothing`). Per the tupleset ruling in Settled: no override semantics, no silent skip.
6. **Tupleset member under-specified.** Traversal met a bare `DirectSubject` member — `folder:x#parent@folder:y`, or equally `@user:anne` — a grammatically legal fact with no resource+`...` slot to substitute the pair's second element into. Distinct code from (5); same family, same upstream Write-path guard.

### Family 3 — the context broke

Infrastructure failures, surfaced through the port and propagated untranslated — the interpreter does not wrap them, since a wrapper adds a layer without adding a fact.

7. **Fact lookup failed.** The pinned port's I/O failed outright.
8. **Snapshot unavailable.** The pin points beyond the retention window (GC took it) or cannot be served — the audit-replay horizon from [[storage-versioning-design]].

### Deliberately absent

- **Schema cycles** (mutual computed subjectsets) — unrepresentable: `Namespace.Create` rejects them at construction, so the evaluator never meets one and Contains carries no schema-cycle guard (ruled 2026-07-18; [[namespace-create-validation]]). The zero-fact recursion graph is the `ComputedSubjectSetRewrite` references, intra-namespace by construction — the type carries only a `RelationshipIdentifier` and roots at the same resource; tupleset arms cannot recurse without consuming a fact, so they count depth instead. Dangling intra-namespace references (computed subjectset targets, tupleset first elements) are rejected in the same pass. The guard lives in construction, not SDL parse, because the Write API builds `Schema` values without touching `Kingo.Sdl`.
- **Timeout / cancellation** — host-edge concern, not a domain error the algebra models.

## Requirements

1. The result is a **`Decision`** (`Kingo.Closures`, stub exists), not a bare bool: the Fact judged, verdict, the snapshot evaluated at, schema version, timestamp — the audit event is the host's request envelope (caller identity et al.) plus this value serialized ([[authz-event-logging]]). The timestamp comes from an injected `System.TimeProvider` (requirement 9's "a clock it didn't receive" — this is the received one). Expand's sibling result type is **`Expansion`** — same five-slot shape, carrying the tree (see Settled; stub exists).
2. Direct membership (`this`) includes **both** direct subject facts **and** the members of subjectset-valued facts stored under the same `SubjectSet` — Zanzibar's userset-expansion clause. A grant to `group:eng#member` must make every member of that set a member here.
3. Contains's walk is **cycle-safe and depth-bounded with a modeled outcome** (a `Result` error, never a crash). Cycles can exist in the graph data (parent loops) — and graph data is user-writable, so an uncrashable hot path is a security property, not a nicety. Schema cycles are unrepresentable: `Namespace.Create` rejects them at construction (see Deliberately absent), so the walk carries no schema-cycle guard. Expand needs no bound: it never recurses (see Settled).
4. Evaluating against an **undefined namespace or relationship is a modeled error**, distinct from a denial. Fail-closed applies to facts, not to schema drift or caller defects.
5. The only I/O is **fact lookup, injected as a port** — the first genuine port family. The port is handed to the interpreter already **snapshot-pinned**: one Check evaluates against one consistent snapshot end to end (the zookie enters the *host API's* signature as an opaque token, consumed at the edge to pin the port — the interpreter's signatures never carry it; [[storage-versioning-design]]). Two implementations by design: Check's cached/hedged one, Read+Expand's plain one.
6. The access patterns this work discovers are the **input to the DynamoDB key design** ([[dynamodblite-substrate]]). Expected at minimum: point lookup (does this fact exist?) and range read (all facts under resource#relationship); record what Expand adds.
7. **Exclusion and intersection semantics are settled — Kleene absorption (see Settled) — and must be pinned by tests.** Evaluation order within them is a cost choice precisely because the semantics are order-free; the SDL example's `banned` exclusion is the obvious fixture, including its error cases (decisive vs absorbed).
8. **Tupleset traversal** (`TupleToSubjectSet`) **is settled — exactly specified traverses, over-/under-specified members are modeled errors (see Settled) — and must be pinned by tests** covering all three member shapes.
   - *Unverified hypothesis, worth testing during design (2026-07-16 — a naming exercise's side effect, not a decision):* the rename to `TupleToSubjectSetRewrite(TuplesetRelationship, ComputedSubjectSetRelationship)` asserts the second component is *the same construct* as `ComputedSubjectSetRewrite`, just rooted at each subject the tupleset walk resolves rather than at this resource — which is Zanzibar's own structure (`computed_userset` nested inside `tuple_to_userset`). If that holds, the `TupleToSubjectSet` arm delegates to the `ComputedSubjectSet` arm per resolved subject instead of reimplementing it, and this requirement's decision is forced rather than chosen: whatever `ComputedSubjectSet` does with a relationship is what the traversal does with it. The thing to check is whether the rooting difference is really the *only* difference — `...` is the obvious place it could break, since a `...` subject has no relationship to compute and may need a distinct arm rather than a delegated one.
9. **Audit emission is an effect at the host edge.** The interpreter returns the `Decision` and never sees a sink, a logger, or a clock it didn't receive (time is an injected `TimeProvider`, per requirement 1).
