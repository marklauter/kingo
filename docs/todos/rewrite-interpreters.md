## The work item

Two evaluators over the `SubjectSetRewrite` algebra, pure domain code in `Kingo.Closures` — the one project that references both `Kingo.Graphs` and `Kingo.Schemas`, and the home of the fact-lookup port its adapters implement later:

- **Contains** — the membership predicate behind the Check API: subject ∈ subjectset? A short-circuiting boolean walk; the hot path.
- **Expand** — materialization of one relationship's rewrite tree, single-level wrt indirection (referenced subjectsets stay leaves; see Settled); the audit/debug/UI view.

This is the first joint consumer of both aggregates — rules from `Spec`, facts from the graph — so expect it to pressure-test the algebra's semantics the way SDL pressure-tested the names ([[the-first-consumer-forges-the-domain]]).

## Naming

- The domain predicate is **`Contains`** — set-first: `subjectSet.Contains(subject)` matches the ∋ reading ([[domain-language]]) and carries .NET's strongest prior (`set.Contains(member)`). Not `IsMember`/`IsAMemberOf`, which read member-first and re-invert that reading.
- **`Check`** stays the service/API-level name — Zanzibar's reach word ([[four-service-split-by-load-profile]]).
- **`Kookie`** is the Kingo name for the zookie (decided 2026-07-16): an opaque kernel value naming a point in the store's timeline. Request-side it's a *floor* ("no earlier than this snapshot"); Decision-side it's the *pin* ("evaluated at this snapshot"). "Zookie" survives only as the paper-facing alias ([[storage-versioning-design]]).
- **The question value is `Fact.SubjectFact`, held putatively** — [[putative-fact]] in the glossary (unified 2026-07-19; supersedes the standalone `Query` type, minted and deleted the same day). A `Fact` is the stored assertion — a closed union over its member's shape: `Fact.SubjectFact` pairs the set with a `SubjectId` member (the `Subject` wrapper record is dissolved — subjects exist only as identifiers inside facts; ruled 2026-07-21, [[resource-fact-case]]), `Fact.SubjectSetFact` with a `SubjectSet` member, and `Fact.ResourceFact` with a bare `Resource` member — the resource itself, the exactly-specified factset shape (third case ruled 2026-07-21; [[resource-fact-case]]). The member seat is named `Subject` in every case — the seat word, the paper's ⟨user⟩. The union moved from the subject seat to the fact itself 2026-07-19, dissolving the `Subject` DU and renaming `DirectSubject` to `Subject`. That refactor dissolved `Query`'s founding rationale — replay from the wide `Subject` DU needed a downcast; `SubjectFact` *is* the refined shape — so the stored-vs-putative distinction survives as stance, not type. The word *query* survives as the seat name: `Decision.Query` is typed `Fact.SubjectFact`.
- **`Kingo.Closures`** is the interpreter project (decided 2026-07-18) — "Acl" names the data structure ReBAC generalized away, and the interpreter answers membership in the derived *closure*: the set closed under the rewrite rules' derivation, graph theory's transitive closure. "Reachability" was rejected: `!` and `&` make the derived set non-monotone, so the verdict is a function over several reachability questions, not a reachability property itself.

## Settled — domain-modeling interviews, 2026-07-16 through 2026-07-18

Kernels pinned with Mark before the clean-room session; treat them as constraints.

- **`Contains` answers one question: does the graph's derived closure contain this fact.** There is no "deny" concept in the algebra — `!` is set exclusion, just another operator. Membership at a snapshot, nothing else; "authorized" is the Check host's interpretation of the answer, not the interpreter's concern. Both truth values are first-class `Decision`s (clarified 2026-07-17): a completed evaluation issues a Decision whether the verdict is true or false — "not included" is the false verdict, fully auditable and replayable, not an absence. "Denied" is the host's reading of a false verdict (and, under fail-closed policy, of a failure — but that denial has no Decision, only the error event).
- **The question is putative and narrowed to check's shape: `Contains(SubjectSet subjectSet, SubjectId subject) → Result<Decision>`** (amended 2026-07-19, supersedes `Contains(Fact fact)`; the party's type is `SubjectId` since the `Subject` wrapper dissolved 2026-07-21, [[resource-fact-case]]). The paper's check takes ⟨object#relation⟩ plus a putative *user* — an authentication token names a direct user, never a userset — and Kingo makes the narrowing type-level: a subjectset-valued question is unrepresentable, not a modeled error. `Fact.SubjectSetFact` is storage vocabulary (facts store subjectset members for userset expansion), not question vocabulary. The shape is closed under recursion: every internal re-entry is itself `(SubjectSet, SubjectId)` — a stored subjectset member re-enters on the *set* side (`Contains(team:sales#member, anne)`), and the subject side never changes type. `Decision` carries the question judged in a seat named `Query`, typed **`Fact.SubjectFact`** — the putative fact (unified 2026-07-19; supersedes the standalone `Query` type — see Naming for the rationale). The case type preserves the narrowing — a `SubjectSetFact` cannot occupy the seat. `Fact.Parse` returns the union, so wherever a question arrives as text the subjectset-subject shape must be refused: the host edge pattern-matches the parse result, or `SubjectFact` grows its own narrower `Parse` — host-edge design work, not settled here. `Contains` reads as an operation on `SubjectSet` — extension-method shape; the mechanics are design work because the evaluation context — the `Spec` value, the snapshot-pinned fact port, the `TimeProvider` — is constructor state, not per-call args. The zookie is consumed at the host edge to pin the port; the interpreter never sees a zookie argument.
- **A subjectset-valued member is an expansion point, never a sought value** (amended 2026-07-19; supersedes the literal-element-membership reading, which existed to give subjectset-valued questions a meaning they no longer need). `Fact.SubjectSetFact` exists so facts can store grants like `doc:x#viewer@group:eng#member`; the walk expands such members per requirement 2 and matches the sought `SubjectId` by value equality at the leaves — no subset reasoning, no second semantics under `!`. A subjectset on the question's subject side is unrepresentable under the amended signature. Matches Zanzibar: check's putative member is a user; Expand's leaves are "user IDs or usersets."
- **`Decision` carries exactly one question — the one judged — and none of the facts consulted to judge it** (amended 2026-07-19; the question value supersedes "the Fact judged"). Decision = the question judged (`SubjectSet` + `SubjectId`), the verdict, the snapshot pin (`Kookie`), the spec version, the wall timestamp. Reproducibility comes from the store's multi-version history, not the payload.
- **The spec version slot is an opaque `SpecVersion` kernel value** (decided 2026-07-18) — minted as a stub (`src/Kingo/SpecVersion.cs`), the same move as `Kookie`; its encoding (hash, revision, name plus revision) is [[storage-versioning-design]]'s to fill. It arrives as evaluator constructor context beside the injected `Spec` value.
- **Empty operator nodes are unrepresentable** (decided 2026-07-19): `UnionRewrite.Create`/`IntersectionRewrite.Create` refuse empty operand lists (`rewrite.union.empty` / `rewrite.intersection.empty`; implemented 2026-07-21, [[namespace-create-validation]]). The SDL grammar cannot produce the shape; the Write API path could, and the conventional reading of an empty intersection — the universal set, everyone a member — is an authorization hazard, so the shape is refused rather than given semantics.
- **The depth bound is evaluator constructor config** (decided 2026-07-16; Contains only — Expand never recurses and takes none, decided 2026-07-18) — injected context alongside the port and clock, not SDL and not per-call. `DepthExceeded` carries the bound it hit, so audit distinguishes "cycle in data" from "bound too low."
- **Operator error semantics are Kleene: the result is a function of operand values, never of evaluation order** (decided 2026-07-16). Error is the third value; an *absorbing* value dominates it: `false` absorbs in `∩` (false & anything = false), `true` absorbs in `∪` (true | anything = true), and for `a ! b` (= `a ∩ ¬b`) both `a = false` and `b = true` absorb to `false`. Short-circuiting is sound exactly when the value in hand is absorbing: the unobserved operand's outcome (true, false, or error) cannot change the answer, so it can be skipped or its in-flight parallel evaluation cancelled. An error surfaces as `Result.Failure` exactly when it is *decisive*: nothing absorbed it, so the answer genuinely depended on the broken operand. The special case already ruled — `a ! b` with `a` true and `b` errored → `Result.Failure`, no Decision — is this rule. Sequential, reversed, and parallel-with-cancellation evaluation all yield the identical verdict; strict error-poisoning was rejected because it is incoherent with short-circuiting (the verdict would depend on scheduling). The residual risk — a broken graph region hiding behind healthy absorbing branches, surfacing only intermittently — is assigned to the Write path and offline validation (spec cycles at construction via `Namespace.Create`, data cycles at write or sweep), not to the hot path.
- **No `Decision.Empty`** (decided 2026-07-17). `Result<Decision>` failure carries no value property at all — the no-answer case is the failure channel itself, so an empty variant has nothing to represent. The general instinct stands elsewhere: where a domain needs a no-answer value, prefer a domain singleton (collection-style `Empty`) over null.
- **Factset traversal: exactly specified traverses; anything else is a modeled error** (decided 2026-07-16). Evaluating `(parent, viewer)` on `folder:x` reads the facts under `folder:x#parent`; each member is one of three shapes — the three `Fact` cases (amended 2026-07-21; [[resource-fact-case]] — case-matching, no `...` sentinel): (1) `folder:y#...` — a `Fact.ResourceFact` member, the bare resource, *exactly specified*: hop 2 pairs the resource with `viewer` and recurses at `folder:y#viewer`; (2) `folder:y#viewer`/`#editor` — a `Fact.SubjectSetFact` member, *over-specified*: modeled error; (3) `folder:y` — a `Fact.SubjectFact` member, a bare subject-id, *under-specified* (no resource to pair the computed relationship with; `folder:x#parent@user:anne` is the same shape — a legal fact, nonsense for traversal): modeled error, distinct code. Both errors join family 2 as ill-shaped-for-traversal data defects. Silent skipping was rejected (authorization outcomes must not change invisibly with data shape); override semantics for (2) were rejected (turns a data oddity into a permission redirect the paper deliberately avoided). The guard belongs upstream: the Write path refuses wrong-shaped members on relationships any rewrite uses as a factset's first element, making the evaluator's error a never-in-practice backstop. That guard is the Write service's work item, not this one; until it exists, conditions 5–6 are the only protection, which is acceptable because they are modeled errors, not crashes.
- **The error taxonomy is settled** (decided 2026-07-16; condition 9 added 2026-07-21) — three families, nine conditions; [Error conditions](#error-conditions) carries the full rulings.
- **Fact/spec drift is closed at the write edge; condition 4 is the backstop** (decided 2026-07-20; dry-run finding F8). Two Write-service invariants. *Facts can't lead the spec*: every fact write is validated against the current spec — namespace defined, relationship defined, factset member shape. *The spec can't abandon facts*: a spec write removing a namespace or relationship is refused while live facts reference it, so removal is a two-step migration — migrate the facts, then land the removal. Whole-namespace and whole-spec deletion are the limiting case; the changelog's deletion entry is a terminal marker, since an empty `Spec` is unrepresentable per `spec.empty`. Evaluation and replay always read the coherent (`Kookie`, `SpecVersion`) pair, both artifacts versioned on the one store timeline. Fact rows carry the interval they existed over (created/tombstoned stamps), a storage artifact below the port and never on `Fact`: stamps are not identity, and grant/revoke/re-grant is one value with two intervals. The spec is an append-only changelog whose version intervals close by supersession. Facts carry no spec version: the abandonment guard makes valid-at-write-time imply valid-under-every-spec-since, so a stored version would be load-bearing only under multi-version evaluation semantics, rejected. Encodings and retention are [[storage-versioning-design]]'s; the invariants are Write-service work items like the conditions 5–6 guard ([[drift-prevention-at-the-write-edges]]); until they exist, condition 4 is the only protection.
- **The pinned port exposes its snapshot token, and the token is the `Kookie` type itself** (decided 2026-07-19) — no distinct port token, no mapping layer. An opaque value the interpreter copies into the `Decision` without interpreting, so the Decision is complete at birth — no host-side enrichment. `Kookie` is minted as an opaque kernel stub (`src/Kingo/Kookie.cs`); encoding is [[storage-versioning-design]]'s to fill.
- **The result channel is `Result<Decision>`** (Contains) **and `Result<Expansion>`** (Expand). Undefined namespace/relationship and depth-exceeded are `Result` failures — where no membership answer exists there is no `Decision`, not a Decision wearing an error flag; likewise where no complete tree exists there is no `Expansion`. Fail-closed (error → deny + alarm) is host policy, visible at the edge. Audit covers both channels: success serializes the Decision; failure serializes the error with the request envelope.
- **`Decision` carries two clocks, distinctly:** the snapshot token (logical time — *which* facts) and the wall timestamp from the injected `TimeProvider` (*when* evaluated). The stamp is taken at evaluation completion (decided 2026-07-20; dry-run finding F16): it records the judgment's own provenance, and the judgment exists when the verdict does — start time is when the question arrived, envelope territory. It stays a plain timestamp, never zookie-encoded: physical time correlates the decision with timelines outside the store (incident windows, token lifetimes, other services' logs); logical time replays it; neither clock derives from the other, and reading audit time out of a `Kookie` would breach its opacity. An investigation uses them in sequence — wall time finds the decisions in a window, each decision's `Kookie` replays its verdict.
- **`Expand` is single-level with respect to indirection — a graph walk of one step, edge choice left to the caller** (decided 2026-07-17; paper §2.4.5 adopted verbatim). One call materializes one relationship's full rewrite tree — the union/intersection/exclusion structure with direct subjects at the leaves — and referenced subjectsets stay unexpanded leaves, a `ComputedSubjectSetRewrite` node included: it materializes as the leaf `resource#relationship`, never as the referenced relationship's inlined subtree (made explicit 2026-07-20; dry-run finding F10). The caller re-expands the leaves it chooses to walk, so transitive closure is the caller's loop, paced edge by edge. Full closure (recurse until only direct subjects remain) was rejected: unbounded fact-driven work per call — the same recursion problem Contains bounds — and resolving `!` leaves drags Expand into re-implementing Contains. A depth-limit parameter was rejected: every consumer must handle partially-expanded trees anyway, so it is the single-level contract with the closure cost profile smuggled in, and it takes whole-level steps where single-level lets the caller decide per edge.
- **Caller identity is envelope, not `Decision`** (decided 2026-07-17). The criterion: a field earns its place on `Decision` iff replaying the decision requires it (Fact, Kookie pin, spec version) or records the judgment's own provenance (verdict, wall timestamp). Caller identity is neither — the same question from a different caller is the same judgment — so it lives in the host's request envelope, and the audit event is envelope + serialized Decision (or envelope + error). The interpreter never sees who asked.
- **`Expand`'s question is a `SubjectSet`** (decided 2026-07-18): `Expand(SubjectSet subjectSet) → Result<Expansion>`. The resource#relationship pair names exactly one relationship's rewrite tree, which is what single-level materializes; no subject in the question — Expand asks "who's in this set," not "is this one in it," so Contains's putative-question shape doesn't transfer. The result is the tree projection: leaves are subject IDs or subjectsets naming other (resource, relationship) pairs; intermediate nodes are union/intersection/exclusion. The optional kookie is the Read+Expand host's request field (the paper's Expand API), consumed at the edge to pin the port — the interpreter sees only the pinned port, same as Contains. Evaluation context (Spec, pinned port, TimeProvider, applicable bounds) is constructor state.
- **`Expand` never recurses; depth exceeded (condition 3) is Contains-only** (decided 2026-07-18; the leaf boundary made explicit 2026-07-20, dry-run finding F10). Expand's unit is one relationship's rewrite rule: it unfolds the queried relationship's operator structure and crosses no reference, schema-side (`ComputedSubjectSetRewrite`) or fact-side (subjectset-valued members, factset substitution results); every reference materializes as an unexpanded leaf. Paper §2.4.5's tree shape settles it: leaf nodes are subject IDs or subjectsets pointing to other (resource, relationship) pairs — if Expand inlined, a subjectset could never be a leaf. "Follows indirect references expressed through userset rewrite rules" is the contrast with Read, which returns only stored facts and cannot see that `viewer` includes `editor`; Expand applies the rewrite rule, so the references become visible as nodes — following means surfacing, not crossing into the referenced definition. Work per call is bounded by schema shape × fact count at one level, never by chain length, so Expand's constructor takes no depth bound.
- **Factset conditions 5–6 apply to `Expand` unchanged** (decided 2026-07-18). The factset arm cannot stay a pure leaf — without reading the facts under `resource#factset-relationship` there is no pair to point a leaf at — so Expand performs the same single fact read Contains does and substitutes to emit leaves; only the recursion into them differs. Same three member shapes, same modeled errors, no silent skip: a tree that silently dropped a parent is the same invisible-outcome hazard the Contains ruling rejected.
- **Condition 4 (undefined namespace or relationship mid-walk) is Contains-only** (decided 2026-07-18; broadened to include namespaces 2026-07-20). It arises exactly when the walk crosses into another relationship's definition; Expand never crosses — both places facts introduce dangling references (subjectset-valued members, factset substitution results) terminate as leaves, emitted as data without schema lookup. A dangling leaf surfaces as family 1 on the caller's re-expand, where it *is* the question. Emitting it is the correct audit behavior: the tree shows what the data contains.
- **Every error `Expand` meets is decisive — whole-call failure, no partial trees** (decided 2026-07-18). Kleene absorption exists because a boolean operand can dominate; a tree has no dominating value — every node is part of the answer, so a broken branch always changes it. Partial trees with error-marker nodes were rejected per the settled channel discipline: where no complete answer exists there is no result value wearing an error flag. Expand's error surface is conditions 1, 2, 5, 6, 7, 8, 9; conditions 3 and 4 are Contains-only.
- **`Expansion` is the sibling result type — same five-slot shape as `Decision`** (decided 2026-07-18). The replay-sufficiency criterion transfers verbatim: `Expansion(SubjectSet, Tree, Kookie, SpecVersion, Timestamp)` — question, answer, pin, spec version, wall timestamp — with `SubjectSet`/tree in the seats the question/verdict occupy on `Decision`. Complete at birth, no host-side enrichment; caller identity stays envelope; two clocks, distinctly. The tree's leaves are subject IDs or subjectsets naming other (resource, relationship) pairs; intermediate nodes are union/intersection/exclusion. The tree DU's shape is design work, not a ruling.
- **The port is async, `ValueTask`-shaped, and the evaluators are async in consequence** (decided 2026-07-20; dry-run finding F11). The port is the domain's own declaration of its IO — the inner edge, drawn in domain vocabulary; the adapter is the outer edge, the same boundary in the infrastructure's vocabulary — and the inner edge has to tell the truth about what crosses it: fact lookup is network storage in the real adapter, and a sync port forces sync-over-async onto the hot path — blocked threadpool threads in a service whose budget is milliseconds at high concurrency. The port is shaped for its worst legitimate adapter, the network one; in-process adapters (cache, test fakes) conform to that shape rather than the reverse, and `ValueTask` makes their synchronous completion nearly free. `Contains` and `Expand` await the port mid-walk, so they return `ValueTask<Result<Decision>>` and `ValueTask<Result<Expansion>>`. `CancellationToken` flows as a parameter through `Contains`, `Expand`, and the port — mechanical plumbing, not domain semantics: cancellation surfaces as an exception at the host edge, never as a modeled error, so the Deliberately absent ruling stands. Not threading the token was rejected: a cancelled request would keep walking the graph and paying for reads nobody wants. Parallel operand evaluation is out of scope: the first implementation is sequential short-circuiting, and the Kleene semantics are order-free precisely so parallelism can be added later without changing a verdict.
- **`Decision` does not carry the proof path.** The witness doesn't survive `!`: a true verdict under `(this | (parent, viewer)) ! banned` embeds a non-membership claim for `banned`, and non-membership has no path — its proof is the exhausted search (a *false* verdict, conversely, can have a concrete path through `banned`). A path also costs hot-path allocation and leaks other subjects' relationships to any caller allowed to ask yes/no questions. "Why" is recomputed on demand: re-run `Contains` or run `Expand` at the Decision's recorded snapshot — reproducibility is a storage property (multi-version facts), not a payload property ([[storage-versioning-design]]).

## Error conditions

Three families, nine conditions. Conditions 1–6 and 9 are distinct modeled errors owned by `Kingo.Closures`; 7–8 are the port's own error values. Every one is a `Result.Failure` — where no answer exists there is no `Decision` (or `Expansion`). In Contains the Kleene rule decides whether a given evaluation surfaces or absorbs it; in Expand every error is decisive. Expand meets conditions 1, 2, 5, 6, 7, 8, 9; conditions 3 and 4 are Contains-only. Condition 9 is numbered by arrival (2026-07-21) and lives in family 2 beside 5–6.

### Family 1 — the question is ill-formed against the schema

Caller defects, detectable before any I/O: the `Spec` is an in-memory value, so both checks are pure lookups that fail before the port is ever touched. Family 1 validates the set side of the question only. The subject side is never schema-validated (ruled 2026-07-19; dry-run finding F9): a `SubjectId` carries nothing schema-addressable — its namespace-shaped prefix is an opaque identifier convention, and identity lives with authn — so there is no lookup to fail.

1. **Undefined namespace.** The question's resource names a namespace the schema doesn't define — `Contains(document:readme#viewer@user:anne)` against a schema that defines `doc`, not `document`. There is no rewrite rule to interpret, so the question isn't answerable *false* — it isn't answerable at all: the set the question names doesn't exist in this schema's universe. A denial asserts "evaluated, and not a member" and is reproducible at the snapshot; this asserts "no such set." Mapping it to `false` would convert every caller typo and every caller/spec version skew into silent denial — an outage that looks like policy.
2. **Undefined relationship on the question.** The namespace exists but the relationship isn't defined on it — `Contains(folder:x#editor@user:anne)` when `folder` defines `owner`/`viewer`/`banned`/`parent` and no `editor`. Same character as (1), kept distinct because the repair is different (wrong namespace vs wrong relationship) and the drift source is different: relationship vocabularies evolve much faster than namespace sets, so the typical cause is a caller compiled against a newer spec than the evaluator holds. This is requirement 4's boundary: fail-closed applies to *facts*, never to spec drift or caller defects. `...` in the question's relationship seat is unrepresentable (ruled 2026-07-21, inverting F18's condition-2-by-contract ruling of 2026-07-20): `...` is not a relationship — the exactly-specified factset member is a `Fact.ResourceFact` carrying a bare `Resource`, and `RelationshipPath` loses the `Nothing` sentinel ([[resource-fact-case]]). `SubjectSet.Parse` refuses `folder:y#...`, so `Contains(folder:y#...@user:anne)` dies at the parse edge and never reaches this condition's schema lookup. The named test moves with the ruling: it pins the parse refusal.

### Family 2 — the evaluation couldn't complete

Defects in the graph, or its interaction with the depth bound. These surface mid-walk, after I/O has begun.

3. **Depth exceeded** (Contains only). The walk hit the constructor-configured bound. The unit is the fact-driven re-entry (ruled 2026-07-18): each factset hop and each subjectset-valued member expansion consumes one unit; `ComputedSubjectSetRewrite` crossings are free — acyclic by construction (see Deliberately absent), they cannot recurse without consuming a fact. The bound cannot be computed at design time because the recursion is not bound by the schema — the schema's rewrite tree is finite and known at parse — it is bound by the *facts*: a factset like `(parent, viewer)` re-enters the walk once per stored parent fact, and subjectset-valued members re-enter it once per nesting level. Chain length is whatever users have written — `folder:a#parent@folder:b`, `folder:b#parent@folder:c`, … arbitrarily long, or `folder:a#parent@folder:b` plus `folder:b#parent@folder:a`, a genuine cycle. The bound is an operational guard against user-writable data; the error carries the bound it hit so audit can distinguish "cycle in data" from "bound set too low."
4. **Undefined namespace or relationship encountered mid-walk** (Contains only — Expand emits the dangling reference as a leaf; broadened 2026-07-20, dry-run finding F8). Same surface as (1)/(2), a different defect: spec/data drift, and the error records which lookup failed, namespace or relationship. Under spec v1, `group:eng#member` is valid and `doc:x#viewer@group:eng#member` is written — validated, stored. Spec v2 removes `member` on `group`, or drops namespace `group` entirely; the fact is still in the store. An evaluation under v2 reaches that member, needs its rewrite, and finds no such relationship (or no such namespace): the fact was valid when written, the spec moved. Factset-resolved resources with undefined namespaces land here too. Both producers of this state are closed at the write edge (see the drift bullet in Settled). With those guards and evaluation always reading the coherent (`Kookie`, `SpecVersion`) pair, this condition is a never-in-practice backstop with the same standing as (5)–(6): reachable through a mismatched pair or a coupling bug, not through normal operation (a pin past the retention horizon fails as condition 8 instead). Treating the dangling reference as an empty set was rejected: it silently changes authorization outcomes under `!`, where an undefined `banned` would mean *nobody is banned*.

5. **Factset member over-specified.** Traversal met a `Fact.SubjectSetFact` member — `folder:x#parent@folder:y#viewer` — where the exactly-specified shape is the `Fact.ResourceFact` member `folder:y#...` ([[resource-fact-case]]). Per the factset ruling in Settled: no override semantics, no silent skip.
6. **Factset member under-specified.** Traversal met a `Fact.SubjectFact` member — `folder:x#parent@folder:y`, or equally `@user:anne` — a grammatically legal fact with no resource to pair the pair's second element with. Distinct code from (5); same family, same upstream Write-path guard.

9. **Member ill-shaped for direct expansion** (ruled 2026-07-21; [[resource-fact-case]]). Direct-membership expansion (`this`, requirement 2) met a `Fact.ResourceFact` member — `doc:x#viewer@folder:y#...` — a resource where a subject or subjectset belongs; the mirror of (6). The shape is meaningful only under factset consumption: the Subject/Resource split makes identity-matching a resource against the sought `SubjectId` a category error, and the paper assigns `#...` no meaning outside factsets. Never skipped, never identity-matched; a resource meant as a direct member is spelled as a subject-id (`doc:x#viewer@folder:y`, a legal `SubjectFact`). Refused upstream by the same Write-path member-shape guard as (5)–(6), same never-in-practice standing. Applies to Contains and Expand alike — Expand meets it materializing `this` members.

### Family 3 — the context broke

Infrastructure failures, surfaced through the port and propagated untranslated — the interpreter does not wrap them, since a wrapper adds a layer without adding a fact.

7. **Fact lookup failed.** The pinned port's I/O failed outright.
8. **Snapshot unavailable.** The pin points beyond the retention window (GC took it) or cannot be served — the audit-replay horizon from [[storage-versioning-design]].

### Deliberately absent

- **Spec cycles** (mutual computed subjectsets) — unrepresentable: `Namespace.Create` rejects them at construction, so the evaluator never meets one and Contains carries no spec-cycle guard (ruled 2026-07-18; [[namespace-create-validation]]). The zero-fact recursion graph is the `ComputedSubjectSetRewrite` references, intra-namespace by construction — the type carries only a `RelationshipPath` and roots at the same resource; factset arms cannot recurse without consuming a fact, so they count depth instead. Dangling intra-namespace references (computed subjectset targets, factset first elements) are rejected in the same pass. The guard lives in construction, not SDL parse, because the Write API builds `Spec` values without touching `Kingo.Sdl`. Implemented 2026-07-21: `Namespace.Create` stages duplicate, dangling-reference, and cycle checks (`namespace.rewrite_cycle` errors carry the full cycle path); [[namespace-create-validation]] records the design. Contains carries no spec-cycle guard.
- **Timeout / cancellation** — host-edge concern, not a domain error the algebra models.

## Requirements

1. The result is a **`Decision`** (`Kingo.Closures`, stub exists), not a bare bool: the question judged (`SubjectSet` + `SubjectId`), verdict, the snapshot evaluated at, spec version, timestamp — the audit event is the host's request envelope (caller identity et al.) plus this value serialized ([[authz-event-logging]]). The timestamp comes from an injected `System.TimeProvider` (requirement 9's "a clock it didn't receive" — this is the received one). Expand's sibling result type is **`Expansion`** — same five-slot shape, carrying the tree (see Settled; stub exists).
2. Direct membership (`this`) includes **both** direct subject facts **and** the members of subjectset-valued facts stored under the same `SubjectSet` — Zanzibar's userset-expansion clause. A grant to `group:eng#member` must make every member of that set a member here. A `Fact.ResourceFact` member here is condition 9 (ruled 2026-07-21; [[resource-fact-case]]).
3. Contains's walk is **cycle-safe and depth-bounded with a modeled outcome** (a `Result` error, never a crash). Cycles can exist in the graph data (parent loops) — and graph data is user-writable, so an uncrashable hot path is a security property. Spec cycles are unrepresentable: `Namespace.Create` rejects them at construction (see Deliberately absent), so the walk carries no spec-cycle guard. Expand needs no bound: it never recurses (see Settled).
4. Evaluating against an **undefined namespace or relationship is a modeled error**, distinct from a denial. Fail-closed applies to facts, not to spec drift or caller defects.
5. The only I/O is **fact lookup, injected as a port** — the first genuine port family. The port is handed to the interpreter already **snapshot-pinned**: one Check evaluates against one consistent snapshot end to end (the zookie enters the *host API's* signature as an opaque token, consumed at the edge to pin the port — the interpreter's signatures never carry it; [[storage-versioning-design]]). Two implementations by design: Check's cached/hedged one, Read+Expand's plain one.
6. The access patterns this work discovers are the **input to the DynamoDB key design** ([[dynamodblite-substrate]]). Expected at minimum: point lookup (does this fact exist?) and range read (all facts under resource#relationship); record what Expand adds in [[fact-store-access-patterns]] — a note seeded when design starts, the deliverable this requirement names (decided 2026-07-20; dry-run finding F19). Whether the port surfaces those as two operations or as one range read the evaluator filters is the clean-room's design choice — this requirement pins the access patterns, not the operation count (assigned 2026-07-20; dry-run finding F20). The evaluator needs no reverse lookup ("which sets contain subject X") — deliberate, confirmed; the only reverse-shaped consumer is the spec-write guard below. The drift ruling adds a cold-path shape outside the evaluator: the spec-write guard's reverse existence query — do any live facts reference this namespace or relationship (dry-run finding F8).
7. **Exclusion and intersection semantics are settled — Kleene absorption (see Settled) — and must be pinned by tests.** Evaluation order within them is a cost choice precisely because the semantics are order-free; the SDL example's `banned` exclusion ([[schema-definition-language]]) is the obvious fixture, including its error cases (decisive vs absorbed).
8. **Factset traversal** (`FactToSubjectSet`) **is settled — exactly specified traverses, over-/under-specified members are modeled errors (see Settled) — and must be pinned by tests** covering all three member shapes.
   - *Unverified hypothesis, worth testing during design (2026-07-16 — a naming exercise's side effect, not a decision):* the rename to `FactToSubjectSetRewrite(FactsetRelationship, ComputedSubjectSetRelationship)` asserts the second component is *the same construct* as `ComputedSubjectSetRewrite`, just rooted at each resource the factset walk resolves rather than at this resource — which is Zanzibar's own structure (`computed_userset` nested inside `tuple_to_userset`). If that holds, the `FactToSubjectSet` arm delegates to the `ComputedSubjectSet` arm per resolved resource instead of reimplementing it, and this requirement's decision is forced rather than chosen: whatever `ComputedSubjectSet` does with a relationship is what the traversal does with it. The thing to check is whether the rooting difference is really the *only* difference — the `Fact.ResourceFact` member is the obvious place it could break, since a bare resource has no relationship of its own and may need a distinct arm rather than a delegated one.
9. **Audit emission is an effect at the host edge.** The interpreter returns the `Decision` and never sees a sink, a logger, or a clock it didn't receive (time is an injected `TimeProvider`, per requirement 1).
